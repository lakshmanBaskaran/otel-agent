Stay rows are the lowest level of data in my system. One stay row means one reservation for one stay date. If a guest stays for four nights, that single reservation will create four stay rows. Reservation count is the number of unique reservations. Room nights are different because they represent occupied inventory and are calculated using the number_of_spaces field. So reservation count, stay rows, and room nights are all measuring slightly different things.

For OTB analysis, I use posted reservations and exclude cancelled reservations by default. Most of the tools work from this filtered dataset. Future OTB metrics are calculated relative to the current anchor date so that only relevant future business is included.

Pickup is based on when a reservation was created, not when the guest is staying. The booking window is calculated using Europe/London time because that's how the hotel business operates. The timestamps are stored in UTC in the database, so the window boundaries are converted to UTC before the query runs.

For market segmentation, I don't rely only on the static macro group stored in market_code_lookup. Instead, I use market_macro_group_history for historical reporting. This lets a market code belong to different macro groups over time and ensures reports use the classification that was valid at that point in time rather than today's classification.
